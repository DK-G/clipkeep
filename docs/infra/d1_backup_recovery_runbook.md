# D1バックアップ / 復旧手順（Runbook）

## 目的
- 障害・誤操作時にDBを復元し、サービス停止時間を最小化する。

## バックアップ方針
- 本番: 日次スナップショット（保持30日）
- 重要リリース前: 手動スナップショット
- 復旧訓練: 月1回（ステージング）

## 事前準備
- `wrangler.toml` の本番DBバインディング確認
- 復旧担当者（Primary/Secondary）を固定
- 復旧時の告知文テンプレを `app/status` と整合

## 復旧フロー
1. 障害検知後、書き込み系APIを一時停止（degraded想定）。
2. 影響範囲（テーブル・期間）を特定。
3. 復元対象スナップショットを選定。
4. ステージングで整合チェック後、本番へ適用。
5. API疎通確認（health / extract / jobs / solution-pages）。
6. statusページ更新、復旧完了告知。

## 検証項目
- 主要エンドポイントの200/400/404/429挙動
- 直近ジョブの整合性（jobId, state, timestamps）
- 参照系ページの表示崩れ有無

## RTO / RPO目標
- RTO: 60分以内
- RPO: 24時間以内
