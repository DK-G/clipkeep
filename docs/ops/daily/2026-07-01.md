## 真因確定（step ① の採取結果）

デプロイ後初の cron `2026-06-30T22:00:20.913Z` が**新コードで書いた heartbeat**:

```json
{"at":"2026-06-30T22:00:20.913Z","discovered":{"twitter":0,"tiktok":0},
 "topicsWritten":0,"jobsLinked":0,
 "diag":{"browserLaunched":false,"tiktokRaw":0,"tiktokKept":0,"twittrendKeywords":0,
   "twitterKept":0,"fallbackUsed":false,"stageErrors":["browser_launch"],
   "browserLaunchError":"Error: Unable to create new browser: code: 429: message: Rate limit exceeded"}}
```

**`browserLaunchError = "Error: Unable to create new browser: code: 429: message: Rate limit exceeded"`**

### step ② 切り分け（4 仮説の棄却/採択）

| 仮説 | 判定 | 根拠 |
|---|---|---|
| binding 欠落 | **棄却** | deploy 出力で `env.browser_rendering → Browser` 付与済み。429（認可済みでレート超過）が返る＝接続自体は成立。 |
| プラン未有効 | **棄却** | 未有効なら 4xx の権限/未有効エラーになる。`Unable to create new browser` の **429 Rate limit** は「有効だが上限到達」を示す。 |
| `@cloudflare/puppeteer` 版不整合 | **棄却** | 版不整合は launch の引数/型エラーになる。サーバ側 429 はランタイム制限であり版の問題ではない。 |
| **日次/同時実行クォータ（レート制限）** | **採択** | 実エラーが明示的に `code: 429: Rate limit exceeded`。Cloudflare Browser Rendering の **ブラウザ新規作成のレート/同時実行/日次上限**に到達している。 |

→ **真因＝Cloudflare Browser Rendering のブラウザ作成レート制限（429）**。anti-bot でも selector 陳腐化でもなく、
そもそも 1 つもブラウザを起こせていない（毎時 `browserLaunched:false`）。

## 残課題＝要人間判断（step ②の是正）

429 の解消は**コードだけでは閉じない＝ダッシュボード/プラン操作が必要**:

- **要確認（Cloudflare ダッシュボード）**: アカウントの Browser Rendering プラン階層・
  **同時ブラウザ上限・日次使用量(分)・新規ブラウザ作成レート**。Workers Free は Browser Rendering の
  同時数/日次が極小で、毎時 cron で容易に 429 到達し得る。
- **判断の選択肢**（人間が決定）:
  1. Browser Rendering を Workers Paid 相当へ引き上げ（課金判断）。
  2. cron 頻度を下げる（毎時→数時間毎）か、**1 ブラウザを再利用**する設計に変更
     （`puppeteer.sessions()` で既存セッションへ `puppeteer.connect`、毎回 `launch` しない）。
  3. ゾンビセッション（close 漏れ）の有無を確認（本コードは finally で `browser.close()` 済みだが、
     launch 失敗時は close 対象が無い＝ゾンビ要因は低い）。
- 上記はいずれも **D1/bindings 非依存**だが、プラン/課金/ダッシュボードの確認・決定が前提のため
  **本日のデプロイ単位では実行せず**、`task.md` 優先1 を「step ②＝要人間判断（プラン/クォータ確認）」へ更新。
