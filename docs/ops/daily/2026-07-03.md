## B. 柱2 browser_rendering 429 の自律是正（option b・無課金）

**背景**: 6/30〜7/01 で真因確定済み＝`Error: Unable to create new browser: 429 Rate limit exceeded`
（本日デプロイ前の heartbeat も `2026-07-02T16:00:20Z` で同一エラー・`browserLaunched:false`）。
従来は step ② を「要人間判断（プラン/クォータ確認）」としていたが、新方針（危害以外は自律）に基づき、
**無課金で完結する option (b) を自律選択**（有料プラン引き上げ=金銭=危害①は選ばず、必要時のみ後日認可依頼）。

**最有力の真因仮説**: 毎時 cron（24回/日）× 1回 60〜120秒のブラウザ時間 ≈ 24〜48分/日 で、
Browser Rendering 無料枠の**日次ブラウザ時間（~10分/日）を早期枯渇**→以降全 run が 429。
（"browserLaunched:false が毎時" という観測と整合）。

**実装（ver `a51db595`, commit 後述）**:
- `src/lib/auto-trend.ts`: 起動前に `puppeteer.limits()` を採取し `DiscoveryDiag` に記録
  （`maxConcurrentSessions`/`activeSessionCount`/`allowedBrowserAcquisitions`/`timeUntilNextAcquisitionMs`）。
  → 429 が「同時セッション上限」か「取得レート/日次予算」かを本番ログ無しに**次回 cron で確定**できる。
- 取得予算ゼロ（`allowedBrowserAcquisitions < 1`）なら **launch せず graceful skip**
  （`acquisitionBudgetZero:true` を記録）＝429 の無駄打ちを停止。
- `src/lib/trends/topic-store.ts`: `DiscoveryDiag` に上記フィールド追加（後方互換・任意）。
- `wrangler.production.toml`: cron を `0 * * * *`（毎時）→ **`0 */6 * * *`（6時間毎, 4回/日）**。
  4回×~90秒 ≈ 6分/日 で日次クォータ内に収める。
  ※ cron 頻度は §6「レート制限」（アプリのリクエスト制限/DO）とは別物＝ガードレール非抵触。

**危害境界の確認**: 無課金（プラン据え置き）・可逆（cron 即戻せる）・D1/bindings/広告ゾーン/レート制限
しきい値 いずれも不変＝危害①〜④に非該当＝自律実行の範囲。

## C. 効果検証（追記・2回目ループ, docs-only）

**注**: 本日の 1 デプロイ枠は B（ver `a51db595`）で消化済み。本節は **1日1タスク厳守** に従い
新規デプロイを行わず、B の 429 是正効果を**検証・記録**するのみ（デプロイ後 §46「次 cron 窓」の被検証）。

**本番確認（再）**: health 200・sitemap 200・`/ja` 200（PowerShell Invoke-WebRequest で確認）。

**取得した本番 heartbeat**（`TREND_KV meta:last_run`, `--remote`）＝**デプロイ後 初の 6時間毎 cron 実行**:
```
at: 2026-07-02T18:00:40.641Z   discovered{twitter:0,tiktok:0}  topicsWritten:0
diag: browserLaunched:false  stageErrors:["browser_launch"]
      maxConcurrentSessions:4  activeSessionCount:0
      allowedBrowserAcquisitions:1  timeUntilNextAcquisitionMs:0
      browserLaunchError: "429: Rate limit exceeded"
```

**判定（真因の再確定）**: 計装した `puppeteer.limits()` が両仮説を**反証**した:
- **同時セッション leak 仮説 → 反証**: `activeSessionCount:0`（orphan なし）。
- **取得予算枯渇仮説（limits が示す範囲）→ 反証**: `allowedBrowserAcquisitions:1` かつ
  `timeUntilNextAcquisitionMs:0`＝limits API は「今すぐ取得可」と申告。にもかかわらず
  実 `puppeteer.launch()` は **429**。
- ∴ 実際の 429 は **`puppeteer.limits()` が露出しないアカウ/プラン水準の Browser Rendering クォータ**
  由来（無料枠の日次ブラウザ分/グローバルレート等）。**6時間毎化だけでは解消を確認できず**、
  graceful-skip ガード（`allowedBrowserAcquisitions<1`）も limits の楽観申告(=1)により**未発火**。

**残る是正の分岐（いずれも別日の枠で・危害境界に留意）**:
1. **（自律・無課金）観測継続**: 翌日の 4 窓（00/06/12/18Z）の heartbeat を確認。1 窓でも非 429 が
   出れば「回転式/一過性レート」で 6h 化が効く見込み。全窓 429 なら無料枠は当該ワークロードで実質不可。
2. **（自律・小・無課金）ガード強化**: 連続 launch-429 を circuit-breaker として計数し、`limits` が 1 でも
   直近 429 実績があれば launch を控える／連続 429 回数を heartbeat に記録（無駄打ち削減・持続性の定量化）。
3. **（要ユーザー判断＝危害① 金銭）**: 無料枠が確定的に不可なら、柱2 discovery の resolution は
   **有料 Browser Rendering プラン** or **非ブラウザ取得（API/RSS 等）への切替**が必要＝ユーザー方向付け要。

**結論**: B の telemetry 計装は機能し、真因を「アカウ/プラン水準クォータ」へ**更新**（leak/取得予算は否定）。
無料枠での完全解消は分岐 3（有料 or 取得方式変更）に至る可能性が高く、これは危害①（金銭）＝**要ユーザー判断**。
次の自律枠では分岐 1（観測継続）＋分岐 2（ガード強化）を先行し、無料枠可否を確定させてから 3 を上申する。
