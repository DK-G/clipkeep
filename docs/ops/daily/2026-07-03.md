## B. 柱2 browser_rendering 429 の自律是正（option b・無課金）

**背景**: 6/30〜7/01 で真因確定済み＝`Error: Unable to create new browser: 429 Rate limit exceeded`
（本日デプロイ前の heartbeat も `2026-07-02T16:00:20Z` で同一エラー・`browserLaunched:false`）。
従来は step ② を「要人間判断（プラン/クォータ確認）」としていたが、新方針（危害以外は自律）に基づき、
**無課金で完結する option (b) を自律選択**（有料プラン引き上げ=金銭=危害①は選ばず、必要時のみ後日認可依頼）。

**最有力の真因仮説**: 毎時 cron（24回/日）× 1回 60〜120秒のブラウザ時間 ≈ 24〜48分/日 で、
Browser Rendering 無料枠の**日次ブラウザ時間（~10分/日）を早期枯渇**→以降全 run が 429。
（"browserLaunched:false が毎時" という観測と整合）。

**実装（ver `a51db595`, commit 後述）**:
- `src/lib/auto-trend.ts`: 起動前に `puppeteer.limits()` を採取し `DiscoveryDiag` に記録
  （`maxConcurrentSessions`/`activeSessionCount`/`allowedBrowserAcquisitions`/`timeUntilNextAcquisitionMs`）。
  → 429 が「同時セッション上限」か「取得レート/日次予算」かを本番ログ無しに**次回 cron で確定**できる。
- 取得予算ゼロ（`allowedBrowserAcquisitions < 1`）なら **launch せず graceful skip**
  （`acquisitionBudgetZero:true` を記録）＝429 の無駄打ちを停止。
- `src/lib/trends/topic-store.ts`: `DiscoveryDiag` に上記フィールド追加（後方互換・任意）。
- `wrangler.production.toml`: cron を `0 * * * *`（毎時）→ **`0 */6 * * *`（6時間毎, 4回/日）**。
  4回×~90秒 ≈ 6分/日 で日次クォータ内に収める。
  ※ cron 頻度は §6「レート制限」（アプリのリクエスト制限/DO）とは別物＝ガードレール非抵触。

**危害境界の確認**: 無課金（プラン据え置き）・可逆（cron 即戻せる）・D1/bindings/広告ゾーン/レート制限
しきい値 いずれも不変＝危害①〜④に非該当＝自律実行の範囲。
