## 真因の確定（コード実体に基づく）

バックログは「`puppeteer.sessions()`+`connect` が stale session へ connect」と推定していたが、
実コード（`src/lib/auto-trend.ts`）は **bare `puppeteer.launch(env.browser_rendering)`** を呼んでいる。
`@cloudflare/puppeteer@1.0.6` の `launch()` は内部で:

1. `acquire(endpoint)` → 新しい `sessionId` を取得（プールが keep-alive の既存セッションを返すこともある）
2. `connect(endpoint, sessionId)` → `connectToCDPBrowser()` が **`Browser.getVersion`** を送信

`connect()` は `connectToCDPBrowser(transport, { sessionId })` としか呼ばず **protocolTimeout を渡さない**ため
`Connection` の**既定 180,000ms（180s）**が使われる。プールが stale/hung なセッションを返すと
`Browser.getVersion` が 180s までハングし、その1本だけで cron が産出ゼロで終わる。エラーは
`connect()` が「Unable to connect to existing session … Browser.getVersion timed out」に包んで再送出する。

**本日デプロイ前の本番 KV heartbeat（baseline, `meta:last_run`）** がこれを裏付け:

```
at: 2026-07-05T18:03:52.674Z
browserLaunched: false / stageErrors: ["browser_launch"]
allowedBrowserAcquisitions: 1 / timeUntilNextAcquisitionMs: 0   ← 取得予算は空き（＝もはや 429 ではない）
browserLaunchError: "Unable to connect to existing session f1bcde8d-7fa0-4ce1-8beb-4c742b80c38b
  (it may still be in use or not ready yet) - retry or launch a new browser:
  tw: Browser.getVersion timed out. Increase the 'protocolTimeout' setting ..."
```

**重要な判断**: 既定が既に 180s のため、バックログ提示①「protocolTimeout 引き上げ」は**逆効果**
（死んだセッションでさらに長くハングするだけ）。正しい是正は**各試行を短時間でバウンドして fail-fast し、
新規 acquire で別セッションに張り直す**（②）＋**確保直後の生存確認**（③）。①は採らない。

## 実装（ver `83afb184-40dc-4f8e-9a5f-1c008889d94c`, commit `7bdfddd`）

- `src/lib/auto-trend.ts`: `launchBrowserWithRetry(binding, diag)` を新設し `discoverTrends` の
  bare `puppeteer.launch` を置換。
  - 最大 **3 試行**。各試行の `puppeteer.launch()` を `withTimeout(..., 30_000ms)` でバウンド（既定 180s より速く打ち切り）。
  - 確保後に `withTimeout(browser.version(), 10_000ms)` で **生存確認**（③。hung セッションはここで速やかに落ちる）。
  - 失敗試行は `browser.close()` で **half-open セッションを破棄**してから次の**新規 acquire**へ（②）。
  - 打ち切り後に遅延解決した launch は自己 `close()` させ**セッションリークを防止**。
- `src/lib/trends/topic-store.ts`: `DiscoveryDiag` に `launchAttempts?`（試行回数）を追加（後方互換・任意）。
- cron 頻度（`0 */6 * * *`）・bindings・D1・広告ゾーン・レート制限しきい値は**いずれも不変**。

**危害境界の確認**: 無課金・可逆・D1/bindings/広告ゾーン/レート制限 いずれも不変＝危害①〜④に非該当＝自律実行の範囲。
