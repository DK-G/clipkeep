## 根本原因（なぜ孤立していたか）
solution ページテンプレート（`src/components/solution-content-client.tsx`）の遷移が **`router.push` の `onClick` ボタン**（戻る導線・CTA）だけで構成されており、**クローラーが辿れる `<a href>` が事実上ゼロ**だった。さらに solution ページ同士を結ぶリンクが一切なく、sitemap には載っていても被リンクが極小 → Googlebot が到達・クロールしづらい構造（impression=0 の主因の一つ）。

## 実装内容
1. **`src/lib/solution-pages/store.ts`**: `getRelatedSolutions(slug, locale, limit=5)` と `RelatedSolution` 型を追加。プラットフォーム別にクラスタリングした内部リンクグラフ（`relatedSlugMap`）＋汎用フォールバック（`defaultRelatedSlugs`）を定義。**全ページが高意図の "not-working" クラスタへ必ずリンク**するよう設計し、新規ロングテールの孤立を解消。タイトルは `findSolutionPage`（ja/pt/ar は en フォールバック）で解決。自身の slug は除外。
2. **`src/components/solution-content-client.tsx`**: `related` prop を受け取り、**クローラー追跡可能な実 `<a href>` の「関連リンク」`<nav>`**（`t.internalLinks` 見出し、各 locale 既存キーを流用）をレンダリング。あわせて CTA・「ホームへ戻る」を `router.push` ボタン → 実 `<a href>` アンカーに変更（`useRouter` 依存を撤去）。locale=en は素のパス、それ以外は `?locale=` 付与。
3. **`app/solution/[slug]/page.tsx`**: サーバ側で `getRelatedSolutions` を計算し client に prop 渡し（SSR HTML に確実に出力＝クローラー可視）。
4. **`app/page.tsx`**: ホームの "Ultimate Guides 2026" グリッドに **Twitter / Telegram の not-working ページを追加**（最高権威ページであるホームからの直接被リンク。既存の Reddit と合わせ 3 本）。
