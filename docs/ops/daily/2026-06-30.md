## 調査（本番 KV `TREND_KV` ns `ab21329d…` を `--remote` で直接確認）

### 真因（2 層）

- **proximate（即効律速）= browser_rendering の puppeteer 起動失敗**: 計装後の本番 heartbeat で
  `browserLaunched:false` / `stageErrors:["browser_launch"]` を確認＝`puppeteer.launch(env.browser_rendering)`
  が throw し、`discoverTrends()` は **どのサイトにも到達せず即 early-return**（L138-142 の catch）。
  つまり「selector の陳腐化」や「anti-bot」ではなく、**ヘッドレスブラウザ自体が起動していない**のが真因。
  TikTok explore / twittrend.jp / Yahoo の各段カウンタが全て 0（`tiktokRaw:0`/`twittrendKeywords:0`）なのも
  起動前に抜けているため。さらに `fallbackUsed:false`＝D1 フォールバック（直近7日の public twitter/tiktok
  完了ジョブ）も 0 件で、no-candidates skip 分岐に落ちている。
  - 想定原因（次イテレーションで切り分け）: Browser Rendering binding が実際には未プロビジョン/プラン制限、
    同時実行/日次クォータ超過、または `@cloudflare/puppeteer` のバージョン非互換のいずれか。
    deploy 出力では `env.browser_rendering → Browser` binding は**付与されている**ため、binding 欠落ではなく
    **ランタイムの起動失敗**。実エラー文字列は現状 `console.error` のみで非永続＝次の計装で `diag` に
    `browserLaunchError`（message 先頭 N 文字）を載せて確定する。
- **structural（潜在・週次レビュー要案件）**: 仮に discover が動いても、現設計では topic が
  `MIN_CLIPS=3` に到達しにくい構造がある（要監視・本日は変更しない）:
  1. **TikTok の topic は常に `null`**（`auto-trend.ts` L162、explore は per-URL の topic ラベルを
     出さない）＝TikTok は topic を一切供給できず、topic ラベルは twitter 経由のみ。
  2. cron は `twitter[0]` / `tiktok[0]` の**各 1 件**しか `allItems` に積まない（L241-242）＝
     1 run あたり 1 topic に 1 clip しか追記されない。
  3. `MIN_CLIPS=3` は dedup 後 3 clip 必要。twittrend のトレンド語は毎時入れ替わるため、
     **同一正規化キーが ≥3 run にわたり別々の新規 URL を供給**する必要があり実現確率が低い。
  - → これは MIN_CLIPS 等の閾値・throughput に関わる**設計判断**で、growth-strategy.md §9 / 2026-06-21
    ユーザー確定により「週次実測で 1 ダイヤルずつ緩める」案件。日次ループで独断変更しない（提案として
    週次 #005 へ記載予定）。

## 実装（今日デプロイできる単位＝計装のみ・挙動非変更）

proximate 律速の是正には「どの段で 0 か」の特定が前提。`meta:last_run` heartbeat に
**段階別診断 `diag`** を追加して、本番ログ無しに discover 破断段を切り分け可能にする。

- `src/lib/trends/topic-store.ts`: `DiscoveryDiag` 型を新設し `RunMeta.diag?`（任意・後方互換）に追加。
  フィールド: `browserLaunched` / `tiktokRaw` / `tiktokKept` / `twittrendKeywords` / `twitterKept` /
  `fallbackUsed` / `stageErrors[]`。
- `src/lib/auto-trend.ts`: `discoverTrends()` に各段のカウンタと段階エラータグ
  （`browser_launch` / `tiktok_scrape` / `twittrend_scrape` / `yahoo_realtime`）を収集して `diag` を返す。
  `runAutoTrendUpdate` の `recordRunMeta` 2 箇所（no-candidates skip 分岐／通常分岐）に `diag` を渡す。
- 既存の topic 永続化・抽出ロジック・ゲートは**一切変更なし**。診断フィールドを KV meta に追記するのみ。
  D1 schema / Cloudflare bindings 不変・新規 URL なし＝sitemap 不変。

切り分けの読み方（次回是正の指針）:
- `browserLaunched:false` → browser_rendering binding / puppeteer 起動の問題。
- `tiktokRaw:0` → TikTok explore のスクレイプ（selector / anti-bot）が破断。
- `twittrendKeywords:0` → twittrend.jp の keyword 取得が破断（最重要＝topic ラベルの唯一供給源）。
- `twittrendKeywords>0` かつ `twitterKept:0` → Yahoo リアルタイム検索段が破断。

## 是正方針（remediation plan）

**確定した真因**: `browser_rendering` の puppeteer 起動失敗（`browserLaunched:false`）。これが
柱2 産出ゼロの**単一・最深の律速**。これが直らない限り、structural 制約の議論は無意味（そもそも
discover が 1 件も返さない）。

次イテレーション（明日以降の日次・各 1 デプロイ単位）:
1. **起動エラーの確定（次の小計装）**: `diag.browserLaunchError`（`puppeteer.launch` の error.message
   先頭 N 文字）を追加デプロイ → 次 cron で実エラーを取得し「未プロビジョン/プラン制限/クォータ超過/
   バージョン非互換」を切り分ける。
2. **Browser Rendering の有効性確認**: Cloudflare ダッシュボード/`wrangler` で Browser Rendering の
   有効化状態・プラン要件・日次クォータ/同時実行上限を確認（**要人間判断**になり得る＝ダッシュボード操作や
   プラン変更が絡む場合は daily log に「要人間判断」と記録して次タスクへ）。`@cloudflare/puppeteer` の
   バージョン整合も確認（HC-4 依存更新と合流可）。
3. **structural 制約は別軸で週次 #005 提案**（browser 修復後に効いてくる）: 1 clip/topic/run・
   TikTok topic null・volatile keyword により `MIN_CLIPS=3` が到達困難。MIN_CLIPS 緩和／1 run の複数 topic
   投入／topic ラベル供給の多重化 を閾値見直し案として起票。日次の独断変更はしない
   （growth-strategy.md §9 / ガードレール遵守）。
